# ページ型タクソノミ

wikiは6種類のページ型で構成する。frontmatterの `type` フィールドで区別する（[frontmatter.md](frontmatter.md)）。

## 圧縮原則（最重要・再掲）

複数ソースを跨いで**圧縮するページ**だけが価値を持つ。1ソースを写しただけのページは負の価値。主役は entity / decision / concept、summaryは来歴として薄く持つ。

- 悪い例: 「2026-07-10定例議事録」の内容をそのままConfluenceページ化する
- 良い例: 定例で出た決定は `decisions/` へ、話題に上がった技術方針は `concepts/` へ、言及された人・システムの最新状態は `entities/` へ反映し、定例そのものは `summaries/` に来歴として一行残すだけ

## entity

プロジェクト・人/役割・チーム・システム・ベンダーなど、実世界の実体を表すページ。**多数のソースを跨いで情報が積み重なる中核ページ**であり、wikiの中で最大の価値を持つ。

- 1entity = 1ファイル。複数ソースで言及されるたびに追記・更新される
- 人物entityは所属・役割・関与プロジェクト・直近の発言/決定への言及を持つ
- システム/ベンダーentityは仕様・契約状況・関連する決定へのリンクを持つ
- 名寄せ（表記ゆれの統合）は必須。詳細は [naming-conventions.md](naming-conventions.md)

```yaml
type: entity
title: 田中太郎
description: XXプロジェクトのテックリード
tags: [人物]
```

## decision

決定・根拠・決定者・日付・撤回/再検討条件・supersede関係を記録する。**チーム用途のキラー機能**（「なぜそう決めたか」を後から追える）。

本文に以下のセクションを必ず含める。

```markdown
## 決定
（何を決めたか、一文で）

## 決定者
（役職/氏名）

## 決定日
（YYYY-MM-DD）

## 根拠
（なぜその決定に至ったか）

## 撤回/再検討条件
（どうなったら見直すか。無ければ「特になし」と明記する）

## 関連
- supersedes: [[旧決定へのリンク]]（あれば）
- superseded_by: [[新決定へのリンク]]（あれば。統廃合されたら記入）
```

`status: superseded` になった決定ページは削除せず、本文冒頭に置き換え先へのリンクを残す。

## concept

横断的なテーマを扱うページ。例: 移行方針、コストガバナンス、セキュリティ方針。

- 複数のentity/decisionを串刺しにする視点を提供する
- 特定の会議・特定のソースに紐づかない、時間を超えて有効な知識を書く

## summary

一次ソース1件 = 1ページ。**軽量に保つ**（来歴 + 要点リンクのみ）。

- 会議議事録・デッキ・チャットスレッド1件ごとに1ファイル
- 本文は長くしない: 日時・参加者・要点3〜5行・関連ページへのリンクのみ
- 詳細な内容は該当する entity/decision/concept 側に反映されているべきで、summaryに二重に書かない

```markdown
## 概要
2026-07-10 の定例。参加者: 田中, 鈴木

## 要点
- [[decisions/2026-07-10-移行方式決定]] を決定
- [[entities/田中太郎]] がテックリードに就任したことを確認

## 出典
raw/transcripts/2026-07-10_定例.md
```

## open_question

未解決の論点・矛盾のキュー。lintのLLM判断による矛盾検出結果もここに流し込む。

- 解消したら `status: superseded` にして、解消の経緯と結論を追記する（削除しない）
- 誰が答えを持っていそうか、次にいつ確認するかを書けると良い

## overview

プロジェクト全体像を示す最上位ページ（`wiki/overview.md` 1本）。index.mdからの入口として、主要なentity/concept/decisionへのポインタをまとめる。
